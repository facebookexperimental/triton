reproducible even after the divergence is fixed. These examples use ONLY stock
@triton.autotune / triton.Config / kernel.warmup() -- no project-local autotuner
changes (equivalence_fn / equivalence_level / artifact_config_prune).

This is the nearest non-"bitequiv:" ancestor of the branch these examples live on
(youngzt/triton-wt-numerical-eg), confirmed an ancestor of
facebookexperimental/triton. To reproduce the exact behavior shown here, build
Triton at this commit and run the examples on a CUDA GPU.

  - GPU:           NVIDIA GB300 (Blackwell, sm_103a, cc 10.3), devgpu013

Note on the stock PyPI wheel: `triton==3.3.1` CANNOT target sm_103a (its bundled
LLVM rejects the processor: "Cannot select intrinsic llvm.nvvm.shfl.sync.bfly").
So on Blackwell you must use the fork build, not the wheel. The divergence itself
is plain upstream behavior (unordered reductions / MMA precision modes), not
fork-specific -- it reproduces anywhere the kernels compile.
