## Known Bugs & Fixes

### 1. getIntOrFloatBitWidth crash on pointer-typed 1D tensors (2026-04-14)
- **Symptom**: `Assertion 'isIntOrFloat()' failed` in `doBufferAllocation`
- **Manifestation**: We hit this when trying to create a 1D channel for pointer tensor. In general, partition scheduler should not put produer and consumer associated with pointer tensor in different partitions. So we will not have a need for a channel that is a pointer tensor. The root cause is in PSM.

### 2. Shared memory overflow from alpha cross-partition channel (2026-04-14, fixed)
- **Symptom**: `OutOfResources: shared memory, Required: 232712, Hardware limit: 232448` in FA forward persistent with dp=2
- **Manifestation**: After rebasing to upstream Triton, `TritonGPURemoveLayoutConversions` chose `#linear` layout instead of `#blocked` for the accumulator. This inserted a `ConvertLayoutOp` between `ExpandDimsOp` and `BroadcastOp` in the alpha correction chain.
- **Fix applied**: Added `cloneOperandChain` in `optimizeSchedule` that walks backward from a cloned `BroadcastOp`/`ExpandDimsOp` and also clones any `ConvertLayoutOp`/`BroadcastOp`/`ExpandDimsOp` feeding it from a different partition.
- **Commit**: `67af25ea`

### 3. optimizeSchedule too broad / too narrow for Blackwell vs Hopper (2026-04-17, fixed)
- **Symptom (Blackwell)**: `channels sharing the same producer must be in the same task` assertion in `WSCodePartition.cpp:createBuffer` when using the broad `isPure(op)` filter.
- **Symptom (Hopper)**: `producerTaskIds.size() == 1` assertion in `CodePartitionUtility.cpp:createChannelPost` when using a restrictive filter that excludes `MemDescTransOp`.
- **Root cause**: The `optimizeSchedule` op filter must be selective:
  - Too broad (any pure single-result op): cascading cloning of expensive ops (`tt.reduce`, `arith.mulf`, etc.) into computation partitions on Blackwell, violating channel invariants.
  - Too narrow (only `ConvertLayoutOp/BroadcastOp/ExpandDimsOp`): `memdesc_trans` shared by two `warp_group_dot` ops in different partitions on Hopper doesn't get cloned, creating a cross-partition memdesc dependency WS can't handle.
- **Fix**: Added `MemDescTransOp` to the allowed op list: `isa<MemDescTransOp, ConvertLayoutOp, BroadcastOp, ExpandDimsOp>(op)`. `MemDescTransOp` is metadata-only (reinterprets shared memory layout) so it's safe and cheap to clone.
- **Lit test**: `partition-scheduling-meta-hopper-fa.mlir` checks for two `memdesc_trans` copies with different partitions.

### 4. Non-deterministic epilogue partition assignment from DenseMap iteration (2026-04-17, fixed)
- **Symptom**: `producerTaskIds.size() == 1` assertion â€” `math.log2` for dp1's result gets partition 2 (dp0's) instead of partition 1, creating a cross-partition dependency with its downstream `arith.addf` in partition 1.
- **Root cause**: Two issues:
  1. Yield operands for `l_i` (softmax sum) and similar non-MMA-feeding ops are NOT in `opToDpId` (they're not in any MMA's backward slice). The post-loop dpId assignment at lines 576-578 skips these results.
  2. The fallback `dpIdToPartition.begin()->second` in `getEpilogueTarget` uses `DenseMap` iteration, which is non-deterministic across builds. Different binaries pick different partitions.
- **Fix**:
  1. Added `findDpIdBackward` helper that walks backward from a yield def through its operand chain to find an ancestor in `opToDpId` (e.g., finds `alpha_exp` which has the correct dpId).
  2. Replaced `dpIdToPartition.begin()->second` with `std::min_element` on the key for deterministic fallback.
- **Lit test**: `partition-scheduling-meta-hopper-fa.mlir` checks that `tt.expand_dims` on `#1` (dp0) gets partition 2 and `#4` (dp1) gets partition 1.

### 5. BWD softmax chain assigned to reduction instead of computation (2026-04-18, fixed)
- **Symptom**: In BWD FA with TMA descriptor_load for m/Di values, the pT chain (`convert_layout â†’ expand_dims â†’ broadcast â†’ arith.subf â†’ math.exp2 â†’ arith.truncf â†’ tmem_alloc`) gets partition 0 (reduction) instead of partition 3 (computation).
- **Root cause**: The load-user scheduling (Phase 4) walks forward from every categorized `descriptor_load` and assigns all transitive users to `defaultPartition`. For BWD, `defaultPartition` falls back to `reductionPartition` (partition 0) via `getDefaultPartition()` since no correction/epilogue/computation partition exists yet. When m/Di values come through `descriptor_load` (TMA), this walk transitively pulls the entire softmax chain into the reduction partition. The lit test used `tt.load` (pointer-based) for m/Di which is NOT categorized as a Load, so the issue was hidden.
- **Fix**: Added guard `defaultPartition != reductionPartition` to the load-user scheduling condition. When `defaultPartition` is just a fallback to reduction (BWD case), the load-user walk is skipped. Phase 5's MMA forward walk correctly assigns the softmax ops to computation instead.
- **Key insight**: The `loops` array in `getInitialSchedule` is ordered `[inner, outer]` (not `[outer, inner]`). Phase 5's `loops[0]` check matches inner-loop MMAs, so `scheduleUsers` DOES run on them. The issue was purely in Phase 4's load-user scheduling being too aggressive.

### 6. Forward set dead-ends at scf.yield inside causal-mask scf.if (2026-05-28, fixed)
- **Symptom**: TMEM exhaustion (`640 vs 512` limit) in causal FA w/ manual dp. The pass produces 8 partitions / 4 computation instead of the expected 6 / 2.
- **Root cause**: In `collectMMABackwardSlices`, the QK MMA's forward user set walk dead-ends at `scf.yield` (a terminator with no SSA results) when it enters the causal-mask `scf.if` region. The forward set is trapped inside the region and never reaches the softmax chain that the PV MMA's backward slice contains. As a result, union-find fails to merge the QK and PV MMAs, so they land in separate computation partitions.
- **Fix**: Make the forward walk follow `scf.yield` â†’ the parent `scf.if`'s corresponding result, using the specific yield operand index so multiple data partitions (e.g., flex attention) are not merged. For multi-result `scf.if`, the walk follows one value at a time from the worklist. With the enlarged forward set, the QK MMA's forward set overlaps the PV MMA's backward slice and they share a compute partition.
- **Lit test**: `partition-scheduling-meta-causal-attention.mlir` (new) checks exactly 6 partitions / 2 computation (fails with 8 / 4 without the fix). Also removed a stale `CHECK-NOT` on `arith.mulf` scores ops in `partition-scheduling-meta-flex-attention.mlir`, which now correctly get computation partitions through the enlarged forward set.

### 7. Empty-producer assertion from partition-less shared `local_alloc` (2026-06-10, fixed)
- **Symptom**: `producerTaskIds.size() == 1` assertion in `CodePartitionUtility.cpp` `createChannelPost` (NDEBUG: `handleOperandD: expected exactly one producer task ID, got 0` + segfault).
- **Manifestation**: FA3 backward (`_attn_bwd_persist`, `cuda:100`, `data_partition_factor=2`). `separateLocalAllocWithSrc` splits a shared `local_alloc` that carries no `ttg.partition` attribute (hoisted above all partitions) into `local_alloc + local_store`; the store inherits the empty partition-id set, so `producerTaskIds` is empty in `createChannelPost`.
- **Root cause**: `createChannelPost`'s `else` branch assumes exactly one producer task. An *empty* set (size 0) â€” not a multi-producer â€” is what trips the assert; the `size() > 1` branch already handles multiple producers.
- **Fix**: Guard `if (producerTaskIds.empty()) return;` â€” an alloc that belongs to no partition needs no cross-partition channel. Companion: `separateLocalAllocWithSrc` now tags the split `local_store` with the source op's single task ID (e.g. the TMA load) so a single-source shared buffer forms a clean 1-producerâ†’N-consumer channel instead of one buffer per consumer.
- **Lit test**: `ws_code_partition_empty_producer_guard.mlir` runs `_attn_bwd_persist` through `--nvgpu-warp-specialization`; aborts without the guard, passes with it.

### 8. Cross-stage SMEM buffer silently downgraded to depth 1 (2026-06-22, fixed)
- **Symptom**: Runtime mbarrier **deadlock** in `_attn_bwd_persist` (FA backward, `ws_persistent`, `cuda:100`) with `early_tma_store_lowering=True` + `num_stages=2` + the heuristic `_BWD_DOT_ATTRS_SCHED` schedule (`bwd_config_idx=1`). Every warp-group spins on `q = desc_q.load` / dependent MMAs.
- **Root cause** (`WSMemoryPlanner.cpp`, `allocateSmemBuffers`): `q` (128Ã—128Ã—f16) is cross-stage â€” consumed by qkT MMA at `loop.stage=0` and dk MMA at `loop.stage=1` â€” so it needs `buffer.copy=2`. Phase 2 set `q`'s 2nd copy then budget-checked via `computeTotalSmem`, which counted the 3 early-TMA staging allocs (dq reduce + dk/dv store, ~32 KB) at full size, so `q`'s 2nd copy exceeded `smem_budget` (200000) and Phase 2 **silently reverted `q` to copy 1**. Phase 3.6 reuse â€” which would alias the staging SMEM and free the space â€” only runs `if (baseTotal > smemBudget)`, but the revert put the total back under budget, so reuse never fired. Chicken-and-egg: reverting the floor kept us under budget, so the space the floor needs was never reclaimed.
- **Fix**: Make the cross-stage minimum a strict, first-applied floor. (1) Phase 2 sets `minCopies = max(maxConsumerStage - minConsumerStage + 1, 1)` (capped at `num_buffers`) via new `getSmemCrossStageDepth`, applied unconditionally with **no budget revert**. (2) Phase 3.6 now reclaims **only discretionary** space â€” TMA-staging buffers â€” and excludes co-live operand buffers (new `isSmemLiveAcrossInnerLoop` guard) so `q`/`k`/`v` are never aliased. (3) If the floor still doesn't fit after staging reuse, ship it anyway (HW SMEM limit / `OutOfResources` at codegen is the backstop) rather than reverting. Phase 4/4.5 reverts/splits clamp to `minCopies`. This realigns the code with the documented design in `SmemAllocationDesign.md` ("no budget check in Phase 2").
- **Lit tests**: `ws_memory_planner_bwd_persist_early_tma.mlir` (new â€” real 128-wide failing IR: `q` now `buffer.copy=2`, store-staging reused). Updated `ws_memory_planner_bwd_persist.mlir` and `ws_memory_planner_bwd.mlir` (`q` 1â†’2, both cross-stage buffers get their floor; `v`/`k` not aliased). E2E: `fused_attention_ws_device_tma.py::test_op[1-False-0-False-triton-fp16-ws_persistent-bwd-False-128-1024-16-8]` now passes (was a hang); `0-â€¦` config unregressed.

### 9. Persistent FA-bwd dv/dk staging SMEM cross-tile race (2026-06-26, fixed)
- **Symptom**: Wrong, **non-deterministic** dv/dk gradients (max-abs error ~3-4 vs ~3e-3) in `_attn_bwd_persist` (`ws_persistent`, `cuda:100`) with `early_tma_store_lowering=True` + `bwd_config_idx=2` (the `_BWD_DOT_ATTRS_TMEM` 3-group reuse config), `HEAD_DIM=128`. The non-persistent (`ws`) path with the same config and the TLX reference are both correct, and persistent idx 0/1/3/4 are correct. compute-sanitizer racecheck flags SMEM hazards on the v/do region (synccheck clean). The combination had **zero test coverage** (`test_op` skipped idx 2 on the persistent path).
- **Root cause** (`WSCodePartition.cpp`, "Step 7.5"): the dv/dk TMA-staging buffers alias the v/do operand SMEM (`allocation.reuseTarget`, realized by `mergeStagingReuseIntoHost`). Across the persistent outer tile loop the *next* tile's operand load can overwrite that SMEM while the *previous* tile's staging TMA store is still reading it. The existing guard was degenerate: a Step-7.5 `producer_acquire` emitted with constant `bufferIdx=0`/`phase=0`, and as a **same-task** channel whose consumer-release is dropped by `WSLowerToken`'s same-partition elision â†’ a no-op. Harmless on the single-tile non-persistent path; a cross-tile SMEM race on persistent.
- **Fix**: Replace the degenerate guard with a dedicated single-buffered **cross-partition** reuse token (mirrors the TLX persistent `k_empties` pattern). The load task `producer_acquire`s it at the top of the persistent outer loop (before the operand loads); the staging task `consumer_release`s it at the bottom (after the staging stores have drained). A separate token avoids the `consumerWarps == nWarps` assert (MMA = 1 warp vs staging = 8 warps). The acquire phase is a 0-based iteration counter normalized as `(iv - lb)/step` (folds to the IV for the normalized persistent loop) so parity is correct regardless of loop bounds/step. The coarse token (acquire-before-all-loads / release-after-all-stores) covers every aliasing pair that shares one outer loop + load task + staging task; the pass verifies that invariant across all `stagingReuseInfos` and skips (with an `LDBG`) rather than emitting a barrier that would guard only one pair.
- **Tests**: Python regression `test_bwd_tmem_dsT_reuse_3group_persistent` (and `test_op` now exercises `ws_persistent`+idx2). E2E: ws_persistent+idx2 now correct (~3e-3) and deterministic (dk/dv run-variance 0); 48-run sweep worst 2.9e-3, 0 NaN. Full WarpSpecialization lit suite: 0 failures. Residual racecheck SMEM hazards are mbarrier false positives (the correct `_attn_fwd_persist` shows the same class).
- **Lit test**: `ws_code_partition_bwd_persist_staging_war.mlir` (real idx2 `_attn_bwd_persist` IR through `--nvgpu-test-ws-code-partition`) pins the Step 7.5 cross-tile WAR token: a single `1x!nvws.token` `producer_acquire`d by the load task (`dstTask=3`) at the top of the persistent outer loop with a loop-carried phase, and `consumer_release`d of the **same** token by the staging task (`dstTask=2`, constant index, region 4) at the bottom â€” i.e. one coarse cross-tile barrier, not per-iteration serialization (the staging is a `memdesc_reinterpret` of the `copy=1` v/do operand, so there is no second slot to pipeline into).

### 10. Early-TMA store-staging: wrong slot + unrealizable-reuse undercount (2026-06-26, fixed; T277224987)
- **Symptom**: Two distinct failures when the FA-bwd early-TMA dk/dv/dq store-staging is driven to depth 2 (e.g. `bwd_config_idx=0/1` at `SMEM_BUDGET=220000`, used as a repro). First, **wrong dv/dk/dq** (~74% mismatch). After fixing that, an **`OutOfResources` (Required 249348 > 232448)** that the planner's own budget model claimed fit (215040 â‰¤ 220000).
- **Root cause A â€” staging slot coupled to outer accumCnt** (`CodePartitionUtility.cpp`, `getStaggeredAccumCnt`): slot was `(accumCnt + theIdx) % numBuffers`. The TMA store-wait is a *fixed in-flight-count* wait (`cp.async.bulk.wait_group(K-1)`, `K = buffer.copy`) that only drains the store issued K stores ago, so it relies on same-slot stores being exactly K apart in issue order. Reuse-group members share one `accumCnt` (+1 per outer iter) and add `theIdx âˆˆ [0,S)`; when `S > K` consecutive iterations' staggered `(slot,phase)` ranges overlap, so the first subtile of tile t+1 reuses the slot the last subtile of tile t just stored â€” reuse distance **1, not K** â€” and `wait_group(K-1)` leaves that store in flight, so the new `local_store` overwrites the slot (or `async_tma_reduce` reads it) before the prior TMA store/reduce drained â†’ wrong dv/dk/dq. Depth-1 (single slot, `wait_group(0)` = fully serialized) is immune, so it only surfaces once Phase 4 double-buffers the staging at the larger budget. (TLX uses `slice_id % DQ_REDUCE_STAGES`.)
- **Root cause B â€” Phase 3.6 marks an encoding-incompatible (unrealizable) reuse** (`WSMemoryPlanner.cpp`, `findReuseCandidate`): the dk/dv store-staging is a 128Ã—32 subtile with **swizzle=64** (`#shared2`); the operands it could alias are 128Ã—128 with **swizzle=128** (`#shared`). `findReuseCandidate` checked byte-fit + liveness but **not encoding**, so it marked the reuse â†’ `computeTotalSmem` excluded the staging (looked free) â†’ planner thought the config fit. But `mergeStagingReuseIntoHost` requires `host.getEncoding() == staging.getEncoding()` (areEncodingsCompatibleForReuse) and silently leaves the staging standalone, so the real footprint blew the HW limit. (This also means bug #8's "store-staging reused to make room" never realized for the 128-wide config â€” it holds purely via the strict cross-stage floor + ship-anyway backstop.)
- **Fix**: (A) `getStaggeredAccumCnt` returns the subtile index directly (`theIdx`, no accumCnt stagger; slot 0 for subtile 0) when the channel's alloc has `buffer.tmaStaging > 0`. (B) new `areReuseEncodingsCompatible` in `findReuseCandidate` rejects targets whose encoding/memory-space/element-type differ (mirrors `areEncodingsCompatibleForReuse`), so Phase 3.6 never marks an unrealizable reuse â†’ `computeTotalSmem` stays honest â†’ Phase 4.5 won't bump the now-counted staging past budget, and `AllocateSharedMemoryNv` packs the honestly-counted buffers under the limit. (Considered a Phase 4.5 `isAllocated` guard too, but it over-aggressively disabled *valid* compatible-encoding copy=2 reuse â€” e.g. idx2's intended double-buffering and `ws_memory_planner_bwd_buffer_reuse.mlir` â€” so it was dropped; the encoding check alone is sufficient and surgical.)
- **Tests**: Updated `ws_memory_planner_bwd_persist_early_tma.mlir` (store-staging now correctly standalone, no `allocation.reuseTarget`/`shareGroup`; q's cross-stage floor still copy=2). New Python regressions `test_bwd_early_tma_staging_depth2` and `test_bwd_early_tma_staging_depth2_persistent` (via a new `test_op(smem_budget=...)` override) force idx0 to `SMEM_BUDGET=220000` so the early-TMA store staging is double-buffered â€” the exact T277224987 repro, which the shipped autotuned matrix never hits (idx0/idx1 pin 200000). E2E: full shipped matrix `test_op` idx {0,1,2} Ã— {ws, ws_persistent} HEAD_DIM=128 all pass; idx2's compatible-encoding reuse still realizes at copy=2. idx0@220000 now both correct and fits: `ttg.shared` â‰ˆ 231940 B < 232448 HW limit, with the dk/dv store-staging allocated **standalone** (0 reuseTarget/reinterpret â€” no stagingâ†”operand space reuse). WarpSpecialization lit suite: 0 failures (90 pass / 11 xfail).
- **Lit test** (root cause A): `ws_code_partition_bwd_staging_slot_rotation.mlir` checks the same-partition dv/dk staging slot is `theIdx % K` reduced (`divui/muli/subi`) from a **constant** subtile index in the staging partition (`ttg.partition = array<i32: 3>`), indexing the `2x128x64xf16` reuse view with `can_rotate_by_buffer_count = 2` â€” versus the loop-carried m/Di buffers, which reduce the `%arg` accumCnt iter_arg. Root cause B (encoding-incompatible reuse) is covered by `ws_memory_planner_bwd_persist_early_tma.mlir`.
- **Follow-up** (T277449158): reclaiming the operand SMEM for the *mismatched*-swizzle case (128x32 sw64 staging vs 128x128 sw128 operand) is deferred â€” neutral/`swizzle=0` backing + per-view reinterpret. `memdesc_reinterpret` cannot convert swizzle (verifier preserves the source encoding), so the current host-typed backing can't express it; and the hostâ†”staging non-overlap is enforced by the bug #9 reuse token, not static liveness, so `AllocateSharedMemoryNv` can't reclaim it on its own. Teaching the allocator about token-enforced disjoint live ranges is explicitly *not* the chosen direction. Full design notes live in T277449158.

### 11. FA-fwd persistent deadlock: bug #10's `getStaggeredAccumCnt` change over-applied to cross-partition staging (2026-06-28, fixed)
- **Symptom**: Runtime **deadlock** (GPU spins at 100%, `cudaStreamSynchronize` never returns) in `_attn_fwd_persist` (FA forward, `ws_persistent`, `fused_attention_ws_device_tma_dp`, non-causal, `cuda:100`/GB200). Compile+launch finishes in ~12s, then the host hangs in `synchronize`/`assert_close`. Introduced by bug #10's commit `cb9d56313`; the parent `27e57b931` is clean (bisected: HEAD~1 passes 4/4, HEAD hangs 3/3). The DP fwd-persistent path *is* covered â€” `test_correctness_autows.py::test_autows_fa_dp_non_causal` (`ws_persistent`, non-causal, Blackwell-gated, listed in `run_all.sh`) was already present at `cb9d56313` and hangs there. The regression escaped not from missing coverage but because that Blackwell-only pytest was not exercised on a GB200 when the bwd-focused #10 landed (its validation was the bwd matrix). Running `test_correctness_autows.py` on a GB200 reproduces the hang and is the regression guard.
- **Root cause** (`CodePartitionUtility.cpp`, `getStaggeredAccumCnt`): bug #10 made the function return the bare subtile index `theIdx` for **every** `buffer.tmaStaging > 0` alloc. That is correct only for **same-partition** staging drained by `cp.async.bulk.wait_group(K-1)` (bwd `dk`/`dv`/`dq`: producer `local_store` and consumer `async_tma_copy`/`async_tma_reduce` are the same warp task). The FA-fwd output staging `desc_o` is *also* `tmaStaging=1` but is **cross-partition**: produced in the compute task (task 0, `local_store`+`arrive(full)`) and TMA-stored in the epilogue-store task (task 2, `wait(full)`â†’`async_tma_copy`â†’`arrive(empty)`) â€” a producer/consumer mbarrier channel. `getBufferIdxAndPhase` derives BOTH the slot AND the empty/full mbarrier phase from the returned count; collapsing it to the constant `theIdx` pins the phase (`wait_barrier ..., %c1_i32`) so it stops flipping across persistent tiles â†’ the producer's empty-wait is never satisfied on tile 1's slot reuse â†’ deadlock. (For `desc_o`, S=K=2 so the *slots* came out right by coincidence; only the phase broke.)
- **Fix**: Gate the bare-`theIdx` path on **same-partition** staging â€” `isSameTaskStaging = isTmaStaging && getWSPartitionIds(ch->getSrcOp()) == getWSPartitionIds(ch->getDstOp())`. Same-partition (wait_group ring) â†’ `theIdx`; cross-partition (mbarrier) â†’ continuous `accumCnt (+theIdx)`, restoring the pre-#10 rotation for `desc_o`. Companion **defensive** change in `WSMemoryPlanner.cpp` `increaseFusedEpilogueCopies`: the same-partition `theIdx % K` rotation is only correct when **K | S** (S = fused subtile count); Phase 4.5 now only bumps a same-partition staging group's copies to depths that divide S. No-op for all shipped configs (`num_stages=2`, `S âˆˆ {2,4}`) â€” it's a backstop against a future config silently violating the invariant. Cross-partition staging is exempt (continuous-accumCnt rotation tolerates any K).
- **Tests**: fwd persistent now correct â€” `test_correctness_autows.py::test_autows_fa_dp_non_causal` passes (was a hang). All 4 bug #9/#10 bwd regressions still pass (`test_bwd_tmem_dsT_reuse_3group{,_persistent}`, `test_bwd_early_tma_staging_depth2{,_persistent}`). WarpSpecialization lit suite: 90 pass / 11 xfail, 0 unexpected failures (unchanged). Docs: `AccumulationCounters.md` (same- vs cross-partition staging table), `TMAStoreWaitPipeline.md` (`K | S` cap).
- **Lit test**: the same-partition (`theIdx`-slot) side of the gate is pinned by `ws_code_partition_bwd_staging_slot_rotation.mlir` (see #10). The cross-partition `desc_o` side (continuous-accumCnt rotation) remains covered E2E by `test_autows_fa_dp_non_causal`; a dedicated fwd-`desc_o` lit checking the rotating empty/full phase is a possible follow-up.
- **Key insight**: `buffer.tmaStaging` alone cannot distinguish the two staging kinds â€” both fwd `desc_o` and bwd `dk/dv` feed `AsyncTMACopyLocalToGlobalOp` and are tagged `tmaStaging=1`. The discriminator is the producer/consumer **task topology** (same vs cross partition), which selects the drain mechanism (wait_group ring vs mbarrier handshake) and hence the slot/phase formula.

### 12. reduce_dq store token-wait misrouted to epilogue â†’ duplicated async_tma_reduce (double-reduce dq) (2026-07-11, fixed)
- **Symptom**: Wrong `dq` gradient (max-abs error ~89 at BN=64; BN=128 correct on launch 0 but wrong from launch 1) in the 2-KV HSTU cross-attention backward (`_hstu_attn_bwd_redq_2kv`, `test_cross_attention_bwd_autows_2kv`) once the 2-KV inner loop is warp-specialized. racecheck/synccheck clean; deterministic; pollution-robust once fixed. T279388065.
- **Root cause** (`PartitionSchedulingMeta.cpp`): the dq store uses TMA-atomic `async_tma_reduce` (`store_reduce="add"`), categorized `TMAReduction` â†’ reduction partition (0). Its completion `async_tma_store_token_wait` consumes the reduce's async token, but `categorizeEpilogueStores` lumped every `TMAStoreTokenWaitOp` into `EpilogueStore`, and the epilogue scheduling walk (`getInitialSchedule`) pulled it into the epilogue-store partition (2, alongside the dk/dv store waits). With the reduce in partition 0 and its wait in partition 2, `doCodePartitionPost` **cloned** the reduce into partition 2 to satisfy the cross-partition token use â†’ **two** `async_tma_reduce` ops â†’ dq reduced into global twice â†’ garbage. The user-visible tell (ttgir2tlx): the same `tma_reduce` appears in two partitions, one standalone (reduction) and one grouped with the `tma_store`s (epilogue).
- **Fix**: `isReduceTokenWait(op)` â€” a `TMAStoreTokenWaitOp` whose token comes from `AsyncTMAReduceOp`/`DescriptorReduceOp`. `categorizeEpilogueStores` categorizes such waits `TMAReduction` (co-locating them with their reduce in the reduction partition), and the epilogue scheduling walk skips them (`isEpilogueStoreOp(op) && !isReduceTokenWait(op)`). Phase 5's TMAReduction scheduling then places the wait in the reduction partition next to its reduce; no clone, one reduce.
